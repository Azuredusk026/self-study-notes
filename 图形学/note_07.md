# 基于物理的渲染（PBR）与光照传播基础

## 一、 核心知识点与原理剖析

### 1. 光源与光线传播基础
*   **光源的本质与距离衰减**：现实中所有光源皆可视为点光源的集合。点光源向周围三维空间发出的总能量恒定，因此随着距离增加，光线照射到包围球体的表面积呈平方（$r^2$）增加。由于能量被平铺在更大的面积上，**单位面积接收到的光能量会随距离的平方成反比衰减**。
*   **方向光（Directional Light）的来源**：当点光源（如太阳）距离受光面（如地球）极远时，由于受光区域在不同高度的球面面积差异微乎其微，距离衰减可忽略不计，且光线方向几乎完全平行同向，此时点光源在数学模型中被近似视为方向光。
*   **光照颜色与几何光学假设**：在现代实时渲染中，通常假定光源只发出红、绿、蓝（RGB）三种波长的光。除微表面波动光学外，传播路径主要基于几何光学（沿直线传播），不考虑衍射。
*   **介质散射效应**：
    *   **均匀介质/真空**：光线沿直线传播，不发生改变。
    *   **瑞利散射（Rayleigh Scattering）**：当介质粒子极小（如大气分子）时，较短波长（蓝光）会被强烈散射到各个方向，而长波长（红光）散射较弱方向偏转小，这是大气颜色与体积云渲染的基础。
    *   **米氏散射（Mie Scattering）**：当粒子大小大于等于光波长（如雾、烟霾）时，对各波长光的散射能力相近，不会改变光的颜色。

### 2. 表面反射与渲染方程
*   **渲染积分方程**：一个表面的出射光亮度 = 该表面的自发光 + **半球空间内所有入射光经过表面反射特性的积分**。实时渲染中不会进行数学上的绝对积分，而是用离散的近似解欺骗人眼。
*   **兰伯特（Lambert）漫反射推导**：入射光横截面面积与受光物体表面积的比值会导致单位受光面积上能量的变化。该比值刚好是入射光线与表面法线夹角的余弦值（$\cos \theta$），即**光线方向与法线方向的点积（Dot Product）**。由于 Lambert 模型对所有观察方向反射强度相同，其自身的双向反射分布函数（BRDF）值为 $1 / \pi$（课程中常把这一整块作为一个基准看待）。
*   **光照现象分类与遮挡处理**：
    *   **直接/局部光照（Local Illumination）**：光线从光源仅发生一次表面反射即进入相机。
    *   **入射光遮挡**：产生阴影。通过**阴影纹理（Shadow Map）**技术实现。若无该技术，背光面也会被错误照亮。
    *   **出射光遮挡**：光线反射向相机的途中被其他物体阻挡。通过**深度缓冲（Depth Buffer）**实现。
    *   **间接/全局光照（Global Illumination）**：发生多次反射后的光照，开销极高（常需光线追踪等技术）。
    *   **次表面散射（Subsurface Scattering）**：光线进入材质内部不均匀介质（如玉石、皮肤），在内部多次反射后再透射出来。普通的漫反射即可理解为一种“局部的次表面散射”。

### 3. 微表面理论与 PBR 核心（Disney Principled BRDF）
*   **微表面模型假设**：无论宏观表面多么粗糙，它都是由海量绝对平滑的“微表面（Microfacet）”组成的。微表面仅遵循几何光学，只发生完美的镜面反射与折射（吸收）。
*   **半角向量（Half Vector, $h$）**：处于入射光方向 $l$ 和视线（出射光）方向 $v$ 夹角正中间的向量。$h = (v + l) / 2$ 并归一化。只有法线刚好等于 $h$ 的微表面，才能将入射光完美反射到视线方向。
*   **BRDF 渲染方程公式**：实时渲染中，出射光最终被拆分为 漫反射（Diffuse） + 高光反射（Specular）。其中高光项公式为：
  $$ \text{Specular} = \frac{D \times F \times G}{4 (n \cdot l)(n \cdot v)} $$
  *公式分母的推导涉及辐射度量学中的立体角积分转换。*

#### D项：法线分布函数（NDF）
*   **作用**：在给定的宏观表面法线 $n$ 下，计算有多少微表面的朝向刚好等于半角向量 $h$。
*   **GGX / Trowbridge-Reitz 分布**：目前最常用的分布模型。控制参数为粗糙度（Roughness）。
*   **粗糙度映射法则**：在 Disney 规范中，代码里实际使用的 $\alpha$ 应该是美术调整的粗糙度参数的**平方**（$\alpha = \text{roughness}^2$），这样能保证视觉上更加线性的过渡。

#### F项：微表面菲涅尔项（Fresnel Reflectance）
*   **作用**：描述绝对平滑的微表面上，发生镜面反射的能量比例（剩余未反射的能量将折射进入内部发生次表面散射/被吸收）。
*   **F0（基础反射率）**：光线垂直打在表面上的反射率。
    *   **绝缘体（电介质）**：F0 极低（如水 0.02，皮肤 0.028，多数物体约 0.04），且**不带颜色**。
    *   **导体（金属）**：F0 较高，且带有金属本身的颜色。
*   **Schlick 近似公式**：无需复杂的波动光学公式，实时渲染采用近似公式：$F = F_0 + (1 - F_0) \times (1 - (h \cdot l))^5$。

#### G项：几何遮蔽函数（Geometry Function）
*   **作用**：微表面之间会互相阻挡光线。分为 **遮蔽（Shadowing）**（阻挡入射光）和 **遮挡（Masking）**（阻挡出射光）。
*   **Smith 高度相关联合函数**：如果单独算遮挡和遮蔽并简单相乘，会导致同一片微表面被计算两次遮挡，画面会变得极暗。因此必须采用 Smith 高度相关的联合遮挡函数（将 $v$ 和 $l$ 共同代入）。

### 4. 漫反射与能量守恒原则
*   **能量守恒逻辑**：入射光打到物体上，**先计算高光反射（F项）**。高光反射用掉的能量，漫反射绝不能再用。因此能够提供给漫反射的能量比例为：$1 - F_{\text{微表面}}$。
*   **金属度（Metallic）的介入**：
    *   纯金属几乎不发生折射后的次表面散射（光透入后被直接吸收产生光电效应），因此**金属没有漫反射**，其颜色表现全靠带有颜色的高光 F0。
    *   通过金属度参数插值：当金属度为 0 时，F0 固定为 0.04，物体颜色（Albedo / BaseColor）进入漫反射；当金属度为 1 时，漫反射能量归 0，物体颜色被赋予给高光 F0。
*   **Lambert漫反射**：最终结合能量守恒的漫反射项代码为：$\text{Diffuse} = (1 - F) \times \text{Albedo} / \pi$。

## 二、 重点术语与概念解析

*   **立体角（Solid Angle）**：以圆锥体衡量视锥范围的概念。像素离相机越远，该立体角对应的物理面积以平方级增大，这刚好与表面发出光线的平方距离衰减相抵消。因此最终写渲染方程进入相机的一步无需再除以距离平方。
*   **宏观菲涅尔效应 vs 菲涅尔反射率**：
    *   *菲涅尔反射率*：物理镜面上真实的反射光与入射光角度关系方程。
    *   *宏观菲涅尔效应*：物体表面凸起处较光滑（受风化摩擦多），凹陷处较粗糙。导致边缘掠角（掠入射）时更容易打中凸起的光滑表面，反射更加强烈统一。
*   **各向异性（Anisotropy）**：微表面结构在特定方向上有规律排列（如头发、拉丝金属），导致产生具有明显方向性拉长的高光形态。
*   **正部函数（$x^+$）**：公式中常见带有加号上标的变量，代表 `max(0, x)`。如果点积小于0则取0。

## 三、 工程经验与避坑指南

### 1. HLSL Shader 编写与计算性能优化
*   **Pow(x, 5) 性能陷阱**：菲涅尔 Schlick 公式中有一个 5 次方计算。直接调用 `pow()` 函数开销大于 5 次单独的乘法。
    *   *优化写法*：`float x2 = x * x; return x2 * x2 * x;` （只需 3 次乘法操作）。
*   **光照计算中的方向约定**：在写代码时，为了统一和方便，**所有的向量（光源方向 $l$、视线方向 $v$、法线方向 $n$、半角向量 $h$）必须统一约定为“从物体表面当前片元出发，指向外部”**。切忌将视线方向写成相机指向片元，否则点积结果全错。相机视线方向代码应为：`CameraPos - WorldPos`。
*   **移动端性能优化（VIS 项）**：完整计算 G 项和分母的 $4(n \cdot l)(n \cdot v)$ 开销较大。移动端常将 G 项与分母约分化简合并为一个名为 `V` (Visibility) 的项进行近似。但必须注意，**简化版在视觉上会明显比完整版更亮**，需根据项目主观容忍度取舍。
*   **数学符号转译错误**：读 PBR 论文的公式转码时极易漏掉括号，或者将最大值取正 `max(0, x)` 误写成了其他操作。公式出错不会报语法错，但视觉效果表现为极其死黑或白爆，排查成本极大。

### 2. RenderDoc 图形管线排错指南
当画面白屏、黑屏或物件丢失时，应遵循以下调试步骤：
1.  **定位 Draw Call (Event Browser)**：在 Event Browser 面板配合 Texture Viewer，从上到下拉动查看到底是哪一步 Draw Call 开始导致画面出错。注意：只有 `Draw Indexed` 等绘制指令才会真正影响画面，前面都是管线设置。
2.  **检查资源绑定 (API Inspector)**：如果某一个 Draw Call 前缺少必要的准备工作（如设置 Render Target，Map Constant Buffer），通过查看 API Inspector 可以看到缺失。建议在引擎代码生成 Buffer 时**使用 Debug Name 命名**，方便抓帧时快速认出对应 Buffer。
3.  **检查管线阶段状态 (Pipeline State)**：
    *   **IA (Input Assembler)**：点开 Buffer 的 Go 按钮，查看顶点坐标、法线、颜色数据是否读入正确。
    *   **VS (Vertex Shader)**：查看 `VS In` 原始网格与 `VS Out`（裁剪空间下的视锥体视图），确认坐标变换矩阵是否乘错导致飞出视锥。
    *   **PS (Pixel Shader) 与深度**：右键点击 Texture Viewer 上的某一个像素，可以查看其具体的颜色值和**深度值**。通过调整界面上的 Depth Range 可以可视化深度缓冲，排查是否被前面的物体挡住。
4.  **CPU 端断点调试兜底**：如果 RenderDoc 中根本抓不到想要的指令（连 Draw Call 都丢了），说明是 CPU 的 C++ 代码分支跑飞了。应对方法是：沿着渲染循环，**每一层级函数打一个断点**，然后 F10 逐行往下走，查看对应的变量（如 Mesh、Material、CommandBuffer容器）内部是否为空，定位是哪一行逻辑没执行进去。
