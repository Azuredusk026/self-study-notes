# 图形学基础与渲染管线核心笔记

## 一、 计算机图形学 vs 图像处理

在行业中，经常有人混淆“图形处理”和“图像处理”，它们的本质数据流向是完全相反的：

* **图形处理 (Computer Graphics, CG)**：**从数据到画面**。输入的是几何体数据（点、线、面、三维坐标），通过图形学算法计算光照、材质，最终在屏幕上渲染出画面（如游戏画面、3D动画）。
* **图像处理 (Image Processing / Computer Vision)**：**从画面到数据**。输入的是现成的图像或视频（如真实拍摄的素材），输出的是提取出的数据或模型（如动作捕捉提取骨骼动画、AI图像识别边缘与特征点）。
* **AI 的冲击**：传统的图像处理是确定性的算法，而现代AI引入了概率模型（大模型/深度学习）。目前AI正在大量侵蚀传统的图形和图像处理岗位，大幅降低了入门门槛并提高了生产效率，这要求图形学开发者必须向更底层的引擎架构或更高阶的算法方向演进。

## 二、 计算机底层数据类型与性能优化原则

在图形学中，性能永远是不充裕的，内存和显存带宽的优化必须精确到字节。

### 1. 核心数据类型精度
* **BYTE**: 8位（常用于颜色通道，如 R, G, B, A 各占一个 BYTE）。
* **short**: 16位整型。
* **int**: 32位整型。
* **half**: 16位浮点数（1个符号位，5个指数位，10个尾数位）。**在 Shader 编程中，能用 half 就绝对不用 float**，以节省GPU寄存器和带宽。
* **float**: 32位单精度浮点数。
* **double**: 64位双精度浮点数（游戏渲染中极少使用，性能开销极大）。

### 2. 内存对齐与优化界限
现代CPU通常是64位架构，底层运算时往往以 32位（4个 BYTE）或 64位 为一次处理块。
因此，处理一个 32 位的 `int` 或 `float`，与处理一个包含 4 个 8 位通道的颜色值（ARGB），在底层的指令耗时上是相近的。
* **优化原则**：数据优化通常做到 32 位（4字节）对齐即可。试图将数据强行压缩到单字节级别进行复杂运算，在现代架构上往往是“负优化”，会导致额外的解包指令开销。
* **颜色运算坑点**：整型颜色值（如 $0 \sim 255$）在底层操作通常是位运算（按位与/或）。如果溢出（如变成256），二进制位会被截断，导致颜色变成纯黑（0）。而在 Shader 中使用浮点数表示颜色时，底层会自动进行 clamp（限制在 $0.0 \sim 1.0$ 之间）。

## 三、 图形学API与基础渲染流程


### 1. 图形API生态与 Shader 的区别
* **图形API (运行在 CPU 主线程/渲染线程)**：DirectX (微软/PC为主)、OpenGL (逐渐淘汰)、OpenGL ES (安卓移动端)、Metal (苹果生态)、Vulkan (跨平台/高性能移动端)。
    * *版本差异*：DX9/DX11 采用的是单线程顺次执行指令的架构；而 DX12 和 Vulkan 支持多线程底层调度，允许多个绘制指令（Draw Call）并行打散提交。
* **Shader (着色器, 运行在 GPU)**：它是跑在显卡上的微型代码段，由图形API（如 DirectX）将数据提交给 GPU 后，由 Shader 负责具体的顶点坐标变换和像素颜色计算。

### 2. DirectX 基础绘制流程 (C++ 代码逻辑解析)
要在屏幕上画一个三角形，必须经历以下严格的管线步骤：

1.  **创建画布 (Create Device)**：调用 API 获取当前系统的图形设备接口。
2.  **创建顶点缓冲区 (Vertex Buffer, VB)**：将内存中的顶点数据（如坐标、颜色）拷贝到显存（Default Pool）或系统内存中。
3.  **定义顶点格式 (FVF - Flexible Vertex Format)**：告诉显卡数据是如何排列的。例如，包含坐标(XYZ)和漫反射颜色(Diffuse)。
4.  **开始绘制 (BeginScene)**：开启当前帧的渲染指令录制。
5.  **清理屏幕 (Clear)**：每次绘制前必须清理上一帧的画面（通常用背景色覆盖）。如果不清理，画面在移动时会产生严重的拖影（锯齿拉伸）。
6.  **绑定数据源 (SetStreamSource)**：将第2步创建的 VB 绑定到渲染管线上。
7.  **触发绘制指令 (DrawPrimitive)**：告诉 GPU 开始画图。每一次调用这个函数，就是一个 **Draw Call**。Draw Call 的数量是衡量 CPU 渲染提交性能的核心指标。
8.  **结束绘制 (EndScene)**：结束指令录制。
9.  **呈现画面 (Present)**：将后台缓冲区（Back Buffer）绘制好的图像交换到前台屏幕上显示。如果没有这一步，玩家什么都看不到。

* **调试避坑**：如果代码没写错但画出来的物体是纯黑的，通常是因为开启了光照状态 (`SetRenderState(D3DRS_LIGHTING, TRUE)`)，但场景里没有放置任何光源。

## 四、 3D空间的核心数学工具

这是图形学程序员的内功，不需要你会解复杂的微积分，但必须深刻理解以下三个数学工具的**几何物理意义**：


### 1. 叉乘 (Cross Product: $u \times v$)
* **运算结果**：得到一个新的**向量**。
* **几何意义**：新向量垂直于 $u$ 和 $v$ 构成的平面。新向量的长度等于 $u$ 和 $v$ 构成的平行四边形的面积。
* **核心用途**：用于计算多边形的**法线 (Normal)**。给定三角形的三个顶点，得出两条边向量，进行叉乘即可得到该三角形朝向哪里的法向量。

### 2. 点乘 (Dot Product: $u \cdot v$)
* **运算结果**：得到一个**标量**（具体的数值）。
* **几何意义**：表示向量 $u$ 在向量 $v$ 方向上的投影长度（当 $v$ 为单位向量时）。
* **核心用途**：用于判断两个方向的夹角关系。如果点乘结果大于0，说明夹角为锐角（方向大致相同）；小于0说明是钝角（方向相反）。这是判断物体**正面还是背面**的核心算法。

### 3. 线性插值 (LERP: Linear Interpolation)
* **公式意义**：在数据 A 和 数据 B 之间，按照比例 $t$（$0 \le t \le 1$）取中间值。
* **核心用途**：图形学中使用频率极高。无论是模型从坐标 A 平滑移动到坐标 B，还是颜色从红色渐变到绿色，或者是骨骼动画的平滑过渡，底层全部依赖 LERP。

### 4. MVP 矩阵变换 (Matrix Transformations)
物体从自身的局部坐标系，最终被画到 2D 屏幕上，需要经过三个核心矩阵的乘法变换：
* **Model/World Matrix (模型/世界矩阵)**：将物体自身的点，摆放到游戏大世界中的具体位置和旋转角度。
* **View Matrix (观察矩阵)**：根据相机的位置、视线朝向 (LookAt) 和上方向 (Up)，把世界坐标转换到相对于相机的坐标系中。
* **Projection Matrix (投影矩阵)**：根据相机的视野范围 (FOV) 和屏幕长宽比 (Aspect Ratio)，将 3D 空间投射为 2D 平面坐标（产生近大远小的透视效果）。

## 五、 场景管理与剔除算法 (Culling)

当你面对一个拥有成千上万物体的庞大场景（如《孤岛惊魂》或《文明》），显卡是不可能把所有东西都画一遍的。必须通过剔除算法，把玩家“看不见”的东西排除掉。


### 1. 背面剔除 (Backface Culling)
一个立方体最多只能同时看到 3 个面，背对着相机的另外 3 个面画出来纯属浪费性能。
* **判定逻辑**：图形API通过顶点的书写顺序（顺时针 CW 或 逆时针 CCW）结合**叉乘**算出面的法线，再用**点乘**与相机视线对比。如果判断为背面，直接丢弃不渲染。
* **调试经验**：如果你发现模型“破洞”或者只显示一半，极大概率是美术导出时法线反了，或者三角形顶点顺序写反了。

### 2. 视椎体剔除 (Frustum Culling)
* **AABB 包围盒**：将一个包含30万面的高模，用一个极简的长方体盒子（AABB）框起来。
* **判定逻辑**：CPU 只需要计算这个极其简单的 AABB 盒子是否在相机的视线范围内。如果盒子都不在视野里，里面的30万个面直接跳过，根本不需要提交给 GPU。

### 3. 遮挡剔除 (Occlusion Culling)
如果一个小人站在一栋巨大的谷仓背后，小人虽然在相机的视野范围内（视椎体剔除失效），但他被完全遮挡，画出来也没用。
* **预计算遮挡查询 (Precomputed Visibility)**：将游戏地图划分为无数个网格区域（Cell）。在打包游戏前，预先计算好“站在 093 号网格里，绝对看不到 013 号网格里的东西”。运行时通过查表直接跳过不可见区域物体的绘制，极大节省性能。
