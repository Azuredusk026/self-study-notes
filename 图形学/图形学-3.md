# 图形学性能优化与渲染管线核心笔记

## 一、 性能优化的核心思想与三大原则

在游戏开发中，性能瓶颈（如掉帧、手机发烫、内存溢出）是永远绕不开的话题。优化的前提是**不能指望硬件升级**，我们必须在现有计算能力下，通过压榨和转移算力来达成目标。

### 1. 优化的三大通用手段
* **改算法（最难但也最根本）**：在数学或数据层面进行精简。
    * *数据降级*：例如将 `float` (32位) 降级为 `half` (16位)；或者将 32 位的完整 RGBA 贴图压缩为 8 位贴图。
    * *数学近似*：放弃完整的 PBR（Physically Based Rendering，基于物理的渲染）光照计算，改用性能开销更小的近似光照算法。
* **空间换时间（预计算）**：将运行时的高昂计算提前算好，存入内存或硬盘中，运行时直接读取。
    * *场景烘焙 (Baking)*：光照、阴影、AO（Ambient Occlusion，环境光遮蔽）全部在打包前渲染成静态贴图，运行时只做贴图采样。
    * *预计算遮挡剔除 (Precomputed Visibility)*：提前算好哪些格子能看见哪些网格，运行时直接查表。
* **时间换空间**：在运行时增加少量计算，以换取极大的存储空间节省。
    * *调色板贴图 (Palette Texture)*：贴图本身不存真实的 RGBA 颜色，而是存一个索引值。运行时在 Shader 中多做一步运算，去调色板数组中查找真实颜色，从而大幅降低贴图占用的显存和包体大小。

### 2. CPU 与 GPU 的算力平衡与运算转移
遇到掉帧（FPS过低）时，必须先通过 Profiler 工具定位是 CPU 瓶颈还是 GPU 瓶颈。
* **GPU 瓶颈（GPU 满载，CPU 闲置）**：将 GPU 的工作挪给 CPU。例如，GPU 非常不擅长处理分支判断（`if/else`），因为 GPU 的架构机制会导致分支两边的代码都被执行一遍。必须将复杂的逻辑判断移到 CPU 中算好，再把确定的结果传给 GPU。
* **CPU 瓶颈（CPU 满载，GPU 闲置）**：将复杂的 CPU 循环挪给 GPU 并行处理。例如判断两个含有大量三角面的模型是否发生物理碰撞。如果用 CPU 写 `for` 循环逐边遍历，开销极大；但如果将每条边的比对任务拆分，分配给 GPU 的成百上千个核心同时进行并行计算，效率会显著提升。

### 3. 总线带宽优化 (Bus Bandwidth)
在手机端（安卓内存大但算力弱，iOS内存小但算力强），**数据传输的总线带宽**往往是隐藏的性能杀手。CPU 与 GPU 之间频繁的全量数据传输会导致设备严重发热和延迟。
* **反面教材**：每一帧在 CPU 算好模型的每一个顶点位移，然后把几十万个顶点全量传给 GPU 渲染。这会瞬间吃满总线带宽。
* **优化方案（骨骼动画/Skinning）**：模型网格（Mesh）预先存放在 GPU 显存中。CPU 每一帧只计算几十根骨骼的变换矩阵（Matrix），将极少量的数据传给 GPU，由 GPU 根据骨骼权重自行计算顶点的新位置。粒子系统（Particle System）同理，只传 Pose，不传全量 Mesh。

## 二、 渲染数据组织与 图形 API 架构

### 1. 主流图形学 API 生态
图形 API 是 CPU 与 GPU 沟通的桥梁。虽然底层硬件一样，但不同平台的调用接口完全不同：
* **PC 端**：DirectX 11 / DirectX 12, OpenGL (逐渐淘汰)。
* **移动端**：OpenGL ES (安卓/iOS 通用基础)，Vulkan (安卓高性能 API)，Metal (苹果专属高性能 API)。
* **通用计算**：CUDA (NVIDIA 专属)。

### 2. DirectX 架构演进与多线程渲染
* **DX9 / DX11**：采用单线程线性渲染管线。数据结构被归一化为 Constant Buffer（常量缓冲区），DX11 开始支持 Streaming Out（回读技术，允许 GPU 算完物理碰撞后把数据返回给 CPU）。
* **DX12 / Vulkan / Metal**：彻底拥抱多线程和高度并行。取消了传统的线性状态机，改用命令队列（Command Queue），允许 CPU 的多个线程同时向 GPU 录制和提交渲染指令。性能天花板极高，但门槛极高，状态锁和内存屏障必须由开发者手动管理。

### 3. 顶点与索引缓冲区 (Vertex Buffer & Index Buffer)
在绘制复杂几何体时，直接使用 `DrawPrimitive` 画三角形会导致大量公共顶点的重复存储和计算。
* **DrawIndexedPrimitive**：现代渲染通常采用索引绘制。先在 Vertex Buffer 中存入所有不重复的顶点数据，再在 Index Buffer 中存入顶点连线的“序号”。这大幅减少了内存占用，并提高了 GPU 的顶点复用率。

## 二、 图形 API 架构与 DirectX 演进

图形 API (Application Programming Interface) 是 CPU 向 GPU 发送指令的接口。虽然底层硬件相同，但不同平台的调用接口完全不同：
* **PC 端**：DirectX 11 / DirectX 12, OpenGL Core (逐渐淘汰)。
* **移动端**：OpenGL ES (安卓/iOS 通用基础)，Vulkan (安卓高性能 API)，Metal (苹果专属高性能 API)。
* **通用计算**：CUDA (NVIDIA 专属，常用于 AI 大模型计算)。

**DirectX 架构演进与多线程渲染**：
* **DX9 / DX11**：采用单线程线性渲染管线。DX11 开始引入 Constant Buffer（常量缓冲区），并支持 Streaming Out（允许 GPU 算完物理碰撞后把数据返回给 CPU）。
* **DX12 / Vulkan / Metal**：彻底拥抱多线程和高度并行。取消了传统的线性状态机，改用指令队列，允许 CPU 的多个线程同时向 GPU 录制和提交渲染指令。性能天花板极高，但门槛极高，各类状态锁必须由开发者手动管理。

**顶点与索引缓冲区 (Vertex Buffer & Index Buffer)**：
在绘制复杂几何体时，直接使用连续绘制会导致大量公共顶点的重复存储。现代渲染通常采用**索引绘制 (DrawIndexedPrimitive)**。先在 Vertex Buffer 中存入所有不重复的顶点数据，再在 Index Buffer 中存入顶点连线的“序号”（通常为 `short` 短整型）。这大幅减少了内存占用，并提高了绘制效率。

## 三、 GPU 微观渲染管线全流程 (Micro Rendering Pipeline)


渲染管线是将 3D 数据最终变成 2D 屏幕画面的核心工作流，分为以下四个严格先后执行的阶段：

### 1. 应用阶段 (Application Stage) -> 运行在 CPU
在 CPU 侧收集所有的模型顶点、贴图资源、Shader 代码，设置渲染状态，最终调用图形 API 发出 **Draw Call**（绘制指令），将数据推向 GPU。

### 2. 几何处理阶段 (Geometry Processing) -> 运行在 GPU
* **顶点着色器 (Vertex Shader, VS)**：这是完全可编程的阶段。针对传入的每一个顶点执行一次 Shader 代码。主要工作是将顶点的局部坐标，乘以 MVP 矩阵变换坐标，并输出后续所需的顶点信息。
* **投影 (Projection)**：将 3D 空间投射为带有深度信息 (Z轴) 的 2D 空间。
* **剪裁 (Clipping)**：  视野（屏幕）之外的图元不需要画。完全在外部的直接丢弃；部分在内部的三角形，会被屏幕边缘“切断”，这会导致原来的一个三角形被切割并重组为多个新的三角形（三角形门）。
* **屏幕映射 (Screen Mapping)**：将剪裁后的标准化坐标（如 0~1 或 -1~1），拉伸映射到实际屏幕的分辨率像素上（比如映射到 1920x1080，长宽比 Aspect Ratio 在此生效）。

### 3. 光栅化阶段 (Rasterization) -> 运行在 GPU 硬件层
此时数据依然是三个点构成的三角形。光栅化器会将三角形覆盖的屏幕像素区域“找出来”。
* 它通过离散的像素网格中心点进行采样，判断该像素中心是否在三角形内部。
* 如果在内部，则生成一个待处理的像素点。
* **锯齿 (Aliasing)** 就是在这个阶段产生的，因为离散的像素网格无法完美表现连续的倾斜边缘。

### 4. 像素处理阶段 (Pixel Processing) -> 运行在 GPU
* **像素着色器 (Pixel Shader, PS / Fragment Shader)**：对光栅化产生的每一个命中点执行一次 Shader 代码。
    * *插值 (Interpolation)*：输入到 PS 中的位置和颜色等数据并不是原封不动的，而是根据该点在三角形内部的位置，由三个顶点的输出数据**线性插值**计算得出的。
    * PS 根据这些数据去采样贴图、计算光照，最终输出一个准确的颜色值。
* **合并/上屏 (Merging / Output Merger)**：计算出的颜色不能直接显示，还必须经过深度测试（判断前后遮挡关系）和混合操作（Blend，处理半透明叠加），最终写入 Frame Buffer（帧缓冲区 Surface），呈现在屏幕上。
