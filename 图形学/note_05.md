# 图形学渲染管线与阴影技术详解 (Forward/Deferred/Shadow Map)

## 一、 核心知识点与原理剖析

### 1. 基础阴影技术：平面映射（“拍饼”操作）
**工作原理：**
将产生阴影的三维模型，沿着光照方向“拍扁”投射到一个给定的二维平面上进行二次绘制。在数学实现上，通常是在 Vertex Shader（顶点着色器）中通过修改投影矩阵或强行修改顶点坐标（例如将 z 轴设为 0，并对 x、y 轴根据光线方向进行偏移拉伸）来实现。

**技术优缺点：**
*   **优点：** 
    *   **极致的性能（省 Draw Call）：** 只需要对原模型更换纯色半透明材质并在地面坐标系下重新绘制一次，仅占用 1 个 Draw Call（DC）。
    *   **绝对硬边无锯齿：** 因为是几何级别的直接投射（三维线段投射到二维依然是平滑的线段），这种阴影边缘极度平滑，没有任何纹理像素采样带来的锯齿问题。
*   **缺点：** 
    *   无法投射到凹凸不平的表面或墙壁上。如果强行要做多平面投射，需要拆分绘制多次，性能成本陡增。
*   **应用场景：** 早期 3D 游戏、对性能要求极高的手游（如早期版本的《阴阳师》、2013-2014年的《英雄联盟》等）。

### 2. 投射阴影（Projective Shadow）
**工作原理：**
在光源位置放置一个专门的摄像机（Light Camera），将场景中特定物体在这个摄像机视角下的图像渲染到一张很小的纹理上（如 256x256 分辨率的 Render Target）。随后，在主视角渲染地面或接收阴影的表面时，把这张纹理当做动态生成的“贴花（Decal）”贴上去。

**技术优缺点：**
*   **优点（性能可控）：** 可以人为打标签，只对需要产生阴影的特定运动物体（如离玩家近的角色）进行投影计算，忽略远景和无关物体，极大地减小了阴影绘制的区域和成本。
*   **平滑处理：** 在将这块生成的贴图贴向地面时，由于纹理拉伸，可以通过**线性采样（Linear Sampling）**或**卷积（Convolution）**提取周边（如 3x3、5x5）像素的加权平均值，从而人为制造出边缘平滑的软阴影效果。
*   **应用场景：** 较老的主机游戏（如《美食从天而降》）处理主角或特定动态物体的局部高质量阴影。

### 3. 渲染目标（Render Target, RT）与离屏渲染
**基础概念：**
在 GPU 的显存中，用于接收渲染输出结果的缓冲区统称为 Render Target。屏幕最终显示的叫 Frame Buffer（帧缓冲，包含 Color Buffer 和 Depth Buffer）。不直接输出到屏幕，而是输出到另一块显存区域的技术称为**离屏渲染（Off-screen Rendering）**。

**典型应用：**
1.  **游戏内视频播放/监视器：** 将视频流（如 H.264 解码后的 Bilibili 视频帧）直接输出到 RT，并作为贴图贴在游戏内的电视机模型上。
2.  **游戏画面推流：** 把当前画面的 RT 截取，编码为 RTMP/RTSP 流推送到直播平台。
3.  **高级渲染管线：** 所有的延迟渲染（Deferred Rendering）、后处理（Post-processing）和阴影贴图（Shadow Map）都重度依赖 RT 技术。

### 4. 核心渲染管线剖析（Render Pipeline）
游戏引擎如何把三维世界画到屏幕上，主要经历了管线的迭代：

#### A. 前向渲染（Forward Rendering）
*   **绘制逻辑：** 按物体一个个画（通常从近到远排序）。对每个物体的每个像素，计算所有打在它身上的光源。
*   **深度测试（Depth Test）：** 为解决被遮挡物体的无效绘制（**Overdraw，重绘浪费**），利用 Depth Buffer 记录屏幕每个像素当前最近的深度值（通常存的是 $1/z$ 以保证精度）。绘制新像素时，对比当前深度，被遮挡则直接丢弃不进行片元（像素）运算。
*   **Pre-Z Pass 优化：** 为了彻底避免 Overdraw（例如大包围盒的房子挡住里面密集的小人），会先不带颜色地把全场景物体画一遍，只更新 Depth Buffer。然后再进行正式的颜色渲染，凡是深度未通过的点统统跳过。
*   **致命缺陷：** 对多光源支持极差。如果场景中有 20 盏灯，每个受影响的物体就要跑 20 次光照代码。通常前向渲染只能支持最多 4 盏实时灯光，其余忽略。

#### B. 前向+ 渲染（Forward+ Rendering）
*   **优化逻辑：** 将原先按光逐个计算，改为“打包计算”。如果一个点受 4 盏光影响，把这 4 盏光的数据打包，一次性送到 Shader 里跑完。
*   **缺陷：** 光源分组通常被强制绑定为特定倍数（如 4 的倍数）。如果物体只受 1 盏光影响，也要走一遍处理 4 盏光的复杂代码，造成运算浪费。同时，过长的 Shader 行数（如 350 行光照计算）容易触发老旧移动端 GPU 的指令数上限。

#### C. 延迟渲染（Deferred Rendering）
*   **绘制逻辑（解决多光源的终极方案）：**
    1.  **几何阶段（Geometry Pass）：** 不计算任何光照，只把场景所有物体的基础颜色（Albedo）、法线（Normal）、粗糙度等物理信息以及深度，分别渲染到多张 RT 上（这被称为 MRT, Multiple Render Targets，统称 G-Buffer）。
    2.  **光照阶段（Lighting Pass）：** 在屏幕空间中，遍历场景里的每一盏灯，根据 G-Buffer 里记录的数据，对每个像素点统一进行一次光照公式计算。
*   **核心优势：** 保证屏幕上的每个像素，无论有多少光源，**只被光照着色一次**。可以轻松支撑同屏几百上千盏灯光（如室内大量的 LED 射灯）。
*   **缺陷：** 半透明物体（如玻璃）无法写入 G-Buffer 被挡住的数据，因此半透明物体必须在延迟渲染结束后，退回到前向渲染的模式再画一遍。

### 5. 现代阴影核心：Shadow Map 与摩尔纹抗锯齿
**工作原理：**
在光源处放相机画一遍场景的深度（记录在 Shadow Map 里）。主相机画场景时，把每个像素的坐标转换到光源空间，取出对应的深度值，去查 Shadow Map 里的深度值。如果像素在光源空间的深度 > Shadow Map 里记录的深度，说明它被挡住了，判定为处于阴影中。

**核心缺陷与解决（阴影粉刺/摩尔纹 Shadow Acne）：**
*   **起因：** Shadow Map 是一张有分辨率上限的贴图（如像素矩阵），深度记录是离散的。当连续的倾斜物理表面去查离散的像素级深度时，部分表面会错误地判定自己“比 Shadow Map 里的值更深”，导致物体表面出现难看的条纹状阴影（自阴影错乱）。
*   **解法（Depth Offset / Near Panel / Bias）：** 在查 Shadow Map 时，人为给查询点向着光源方向偏移一段距离（Offset）。
*   **精度极限：** Unity 中这个偏移量通常是 `0.1 unit` 或是光照范围的 `1%`。这源于 Shadow Map 贴图本身的精度极限（例如最低标准 15-bit 深度格式，最多区分 $2^{10}$ 即 1024 级，精度就是 $1/1024 \approx 0.001$）。

### 6. 软阴影算法演进（Soft Shadows）
为了消除 Shadow Map 投影在边缘产生的锯齿硬边，采用了以下算法：
*   **PCF (Percentage-Closer Filtering)：** 在 Shadow Map 判断阴影时，不只查中心 1 个点，而是向周围多次采样（如 3x3=9次，5x5=25次）并取加权平均值，把硬边强行模糊成渐变的软边。
    *   **缺点：** 违背物理直觉。真实的阴影是距离遮挡物越近边缘越硬，越远色散越厉害边缘越柔。PCF 无论阴影远近，模糊的范围是一样宽的。
*   **PCSS (Percentage-Closer Soft Shadows)：** 百分比近似软阴影。它先在 Shadow Map 里搜索遮挡物（Blocker）的平均深度，算出遮挡物离当前接收阴影表面的距离。**距离越远，PCF 采样的搜索范围（Filter Size）就拉得越大，阴影越柔和**，完美模拟了真实世界的光学散射。


## 二、 重点术语与概念解析

*   **Draw Call (DC):** CPU 向 GPU 发送的一次绘制指令。DC 数量越低，CPU 开销越小，游戏越流畅。
*   **AABB (Axis-Aligned Bounding Box):** 轴对称包围盒。不随物体旋转而旋转的正交包围盒，渲染引擎最常用来做视锥体剔除和深度排序的简单几何体。
*   **Overdraw:** 重绘。屏幕上同一个像素点被后面的物体反复覆盖绘制了多次，导致 GPU 填充率浪费。
*   **Z-Buffer / Depth Buffer:** 深度缓冲。RT 的一种，专门记录像素离相机的距离（通常存储 $1/z$），用于深度测试以解决遮挡关系。
*   **MRT (Multiple Render Targets):** 多目标渲染。允许 GPU 一次运算同时往多张纹理（RT）上写入不同数据，是延迟渲染（Deferred Rendering）的基石。

## 三、 工程经验与避坑指南

1.  **关于深度值（Depth）存储的数学细节：**
    深度缓冲几乎不会直接存纯距离（Z值）。因为距离的范围可能达到数十公里，如果用常见的 16-bit 浮点数存纯 Z，远处的精度会严重丢失导致画面闪烁（Z-fighting）。所以工业界普遍存储的是反深度 **$1/z$**，这样可以确保所有的值都被规范化为小数，且近处的精度极高。
2.  **Forward+ 的性能陷阱：**
    不要迷信新技术。如果你的游戏场景绝大部分时候只有一两盏光源（如只有太阳光的户外大世界），强行开启 Forward+ 可能会因为 Shader 里的分支和循环代码（为凑齐4倍数处理）导致性能反降，老老实实用标准的前向渲染即可。
3.  **调试工具避坑：**
    老师在课程中使用的图形抓帧分析工具 **IGPA (Intel Graphics Performance Analyzers)** 在接入最新版 Unity (如 Unity 2025) 时存在严重的底层库崩溃/不兼容问题（可能与近期的渲染引擎底层安全漏洞升级有关）。若要进行稳定的图形管线 Debug，建议回退使用 **IGPA 2019 或 2022 稳定版**，或改用侵入式抓帧工具如 RenderDoc。
