# 渲染管线与宏观渲染架构核心笔记

## 一、 微观渲染细节：剪裁 (Clipping) 算法解析

在上节课的作业中，我们留了一个问题：当一个三角形超出屏幕边界时，GPU 底层是如何将其“剪裁”的？

### 1. 线段的剪裁伪代码逻辑
当一条线段穿过屏幕边界时，计算机会执行以下步骤：
* **步骤 1 (边界碰撞检测)**：将线段的两个端点 $(X_0, X_1)$ 与屏幕边界（如 $X=0.5$）进行对比。如果一个大于边界值，一个小于边界值，说明线段被该边界切断。
* **步骤 2 (交点计算)**：利用线性插值（或直线方程），计算出线段与屏幕边界的交点 $P_{new}$。
* **步骤 3 (数据替换)**：丢弃处于屏幕外部的端点，保留屏幕内部的端点。将内部端点与新算出的交点 $P_{new}$ 连接，形成新的线段。

### 2. 三角形的剪裁与重组 (三角形门)
三角形是由三条边构成的。当三角形的一个角超出屏幕时：
* 必定有**两条边**与屏幕边界相交，产生两个新的交点（$O_1, O_2$）。
* 外部的顶点被丢弃后，原本的一个三角形（如 $O, P, Q$）会被这两个新交点“切”成一个四边形。
* **核心动作**：由于 GPU 只认三角形，它会将这个四边形沿对角线一分为二，重组成**两个新的三角形**（如 $\triangle O_1 P Q$ 和 $\triangle O_1 Q O_2$）。
* *极端情况*：如果一个三角形压在了屏幕角落的十字边界上，它可能会被切碎重组成 3 个甚至 4 个全新的三角形输入给后续的光栅化阶段。

## 二、 宏观渲染机制：帧率、多线程与 Buffer 交换

当所有像素的颜色都被计算完毕后，GPU 是如何把它们画到屏幕上的？

### 1. 原子操作与双缓冲机制 (Double Buffering)
GPU 计算每一个像素点颜色的速度是不一致的（有的大招特效算得很慢）。如果 GPU 算一个点就往屏幕上画一个点，玩家就会看到画面在逐行刷新或撕裂（Screen Tearing）。
* **Back Buffer (后台缓冲)**：GPU 会在后台内存中准备一张“隐形的画布”。
* **原子操作**：GPU 必须等待当前帧**所有的点**都画满了这张隐形画布，才算一帧渲染结束。
* **Swap Buffer (交换缓冲)**：当前帧画完后，GPU 将后台缓冲 (Back Buffer) 与前台屏幕正在显示的缓冲 (Front Buffer) 瞬间交换。此时完整的画面才会呈现在玩家眼中。

### 2. 帧率波动的本质与屏幕撕裂
* **理想情况 (60Hz屏幕与60FPS游戏同步)**：屏幕每刷新 1 次，GPU 刚好算完 1 帧，完美对应。
* **掉帧/毛刺卡顿**：如果 GPU 算到某一帧时（比如同屏突然出现了 600 个半透明叠加爆炸），计算量陡增，导致这一帧花了 3 个屏幕刷新周期才算完。那么屏幕在这 3 个周期内只能被迫显示上一帧的旧画面，这就是玩家感受到的“卡顿”。
* **三缓冲 (Triple Buffering)**：为了缓解上述等待问题，引擎会开启三个 Buffer。当 Front Buffer 在显示，Back Buffer1 被卡住还没画完时，GPU 可以提前在 Back Buffer2 中去画下一帧的数据，从而减少 GPU 的闲置等待时间。

## 三、 主线程与渲染线程的解耦机制 (格斗游戏判定逻辑)

在现代游戏架构中，**逻辑运算 (CPU)** 与 **画面渲染 (GPU)** 往往是不在一个线程且不同步的。

### 以格斗游戏（如拳皇/街霸）的打击判定为例：
* **逻辑帧 (Logic Frame)**：格斗游戏的打击判定极其严苛（通常固定为 60 帧/秒）。当玩家按下攻击键，系统会在代码层面的特定某一帧，立刻生成一个“攻击碰撞框 (Hitbox)”，去和对手的“受击框 (Hurtbox)”比对。此时虽然产生了“命中”的结果，但玩家此时在屏幕上**根本还没看到拳头挥出去**。
* **渲染帧 (Render Frame)**：逻辑判定命中后，CPU 将“播放受击动画”、“生成血液特效”等指令打包发送给 GPU 的渲染指令队列 (Command Buffer)。GPU 收到指令后，经过复杂的计算，最终在屏幕上画出画面。
* **延迟本质**：从玩家按下按键（逻辑生效），到屏幕上真正显示出“拳头打中人”的画面，中间通常会间隔 1~3 个渲染帧的物理延迟。

## 四、 抓帧实战：2D 游戏的两种渲染流派

通过 Intel GPA 抓取商业游戏的渲染管线，老师演示了两种完全不同的 2D 游戏渲染逻辑：

### 1. 传统的 2D 游戏渲染技巧（画家算法）
* 传统 2D 游戏（如早期的贪吃蛇或卡牌UI界面）没有 3D 的深度（Z轴）概念。
* 渲染管线把所有 2D 精灵图（Sprite）视为半透明物体。
* **画家算法**：必须严格按照**从远到近（从后往前）**的顺序一层层画。先画最远处的背景底色，再画山，再画树，最后画最上层的 UI 和人物。这会导致严重的 **Overdraw（重绘浪费）** 问题，因为远处的山会被近处的人遮挡，但远处的山还是被完整计算并画了一遍。

### 2. 现代 2D 游戏渲染技巧（借助 3D 深度缓冲剔除）
在老师抓取的《天天炫斗》渲染分析中，发现了开发者利用 3D 引擎技术优化 2D 渲染的高级技巧：
* 虽然游戏画面是 2D 面片，但开发组为每一个 2D 图层赋予了 3D 坐标的**深度值 (Z-Depth)**。
* **渲染顺序反转**：渲染时采用**从近到远**的顺序（先画最前面的 UI 和人物，最后画最远处的背景）。
* **深度测试 (Depth Test)**：当画到最远处的背景时，GPU 会检查当前像素。如果该像素已经被近处的人物“霸占”了深度值（深度测试未通过），GPU 就会**直接丢弃**背景上该像素的绘制计算（这一技术被称为 Early-Z Culling）。
* 这种方式极大节省了像素着色器（Pixel Shader）计算被遮挡物体的性能开销。
