# 图形学与渲染基础入门

## 一、 核心知识点与原理剖析

### 1. 图形处理与图像处理的区别
*   **图形处理 (Computer Graphics)**：核心路径是“数据 -> 图像”。从基础数据（空间中的点、线、面），组成三角面阵列（模型），再摆放成场景，最终通过光栅化或光线追踪绘制在屏幕上表达内容逻辑。
*   **图像处理 (Image Processing)**：核心路径是“图像 -> 数据”。从原始图像数据出发，进行边缘分析、特征识别，甚至生成新的图像。
*   *AI 的介入差异*：传统的图像处理是“真实数据到真实结果”，而生成式 AI 则是“真实数据到概率生成的看似真实的结果”，两者存在准确性和概率扰动的本质区别。

### 2. 基础数据类型与内存运算
*   **数据类型与精度**：`BYTE` (1字节), `short` (2字节), `int` (4字节), `half` (2字节, 16位浮点), `float` (4字节, 32位浮点), `double` (8字节, 64位浮点)。
*   **64位机器运算逻辑**：64位CPU一次通常可以处理4个`BYTE`级别的运算（组合成32位指令进行调度）。
*   **性能优化原则**：在Shader或底层运算优化中，优化到 `32位`（4个BYTE）级别的运算即为有效优化的极限。若尝试继续向下拆分优化到单个 `BYTE`，往往会变成“负优化”。

### 3. 图形学底层 API 与 Shader
*   **图形学 API**：运行在 **CPU** 端的调用接口。如 DirectX (DX), OpenGL, OpenGL ES (移动端), Metal (苹果), Vulkan。
*   **Shader（着色器）**：运行在 **GPU** 端的一小段代码，用于处理顶点变换或像素颜色。
*   **架构演进**：
    *   **DX9 / DX11**：采用串行或多线程指令顺次执行机制（即画完一笔再画下一笔）。
    *   **DX12 / Vulkan / Metal**：采用完全并行、打散的混合渲染策略，虽然难以直观理解，但性能极高。

### 4. 渲染管线与绘制流程
在图形API（以 DirectX 9 为例）中，渲染一个物体的基本流程如下：
1.  **准备画布**：通过 `CreateWindow` 创建系统窗口，随后 `CreateDevice` 创建 DirectX 渲染设备上下文。使用 `ShowWindow` 和 `UpdateWindow` 将其展示。
2.  **准备数据 (Init VB)**：创建顶点缓冲区 (Vertex Buffer)。定义顶点的数据格式 (FVF，如位置`XYZ`、颜色`Diffuse`等)，并在内存或显存 (Default/System Memory) 中锁定 (Lock) 内存，将顶点数组 Copy 进去后再解锁 (Unlock)。
3.  **渲染循环 (Render Loop)**：每帧执行的核心逻辑：
    *   `Clear`：清空背景缓冲区（设置背景色）。
    *   `BeginScene`：开启当前帧的场景绘制。
    *   `SetStreamSource`：绑定之前准备好的顶点数据流。
    *   `SetFVF`：告知 GPU 顶点的解析格式。
    *   `DrawPrimitive`：执行图元绘制（发出 Draw Call）。
    *   `EndScene`：结束场景绘制。
    *   `Present`：将后台缓冲区 (Back Buffer) 的画面交换并呈现到屏幕上。

### 5. 基本图元 (Primitives) 与绘制方式
*   **点 (Point List)**：单独的顶点。
*   **线 (Line)**：分为 `Line List`（两两配对成线）和 `Line Strip`（首尾相连的线带）。
*   **三角形 (Triangle)**：分为 `Triangle List`（每3个点独立成面，最常用）、`Triangle Strip`（共用边的三角带）和 `Triangle Fan`（共用中心点的三角扇）。
*   *注*：DirectX 中没有独立的四边形图元，四边形一律由两个三角形拼接而成。

### 6. 图形学核心数学基础
图形学极度依赖线性代数，主要涉及 2D/3D/4D 向量运算及矩阵变换：
*   **向量点乘 (Dot Product)**：结果为一个**标量**。
    *   *几何意义*：向量 A 在向量 B 上的投影长度（若 B 为单位向量）。
    *   *核心应用*：判断两个向量的朝向关系。常用于**背面剔除 (Backface Culling)**。将视线向量与三角面法线向量进行点乘，结果为正或为负决定了我们看到的是三角面的正面还是反面。
*   **向量叉乘 (Cross Product)**：结果为一个**向量**。
    *   *几何意义*：生成一个垂直于这两个向量所在平面的新向量，其长度等于这两个向量构成的平行四边形的有向面积。
    *   *核心应用*：用于求取三角面的**法线向量 (Normal)**。
*   **线性插值 (Lerp)**：依据比例 `t` (0~1) 在两个数值（位置、颜色等）之间进行平滑过渡求值。
*   **矩阵变换 (MVP)**：
    *   `Model (World) Matrix`：将顶点从模型局部坐标系转换到世界坐标系。
    *   `View Matrix`：由相机位置、观察目标点、向上方向 (Up Vector) 决定，将世界坐标转为相机观察坐标。
    *   `Projection Matrix`：由视场角 (FOV) 和屏幕长宽比 (Aspect) 决定，进行透视投影。
    *   *运算顺序*：通常为 `顶点 × World × View × Projection`，或取决于具体引擎的左乘/右乘规则。

## 二、 重点术语与概念解析

*   **AABB Box (Axis-Aligned Bounding Box，轴对齐包围盒)**：一种用于包裹三维模型的三维长方体，其边平行于坐标轴。常用于预先的碰撞检测和遮挡查询，以极大节省对内部数万个三角面进行计算的性能。
*   **Draw Call**：CPU 准备好数据并向 GPU 发出的“画一次”的指令。无论这一笔画了多少个点，调用一次 API 就算作一个 Draw Call。这是衡量渲染性能开销的关键指标。
*   **FOV (Field of View，视场角)**：决定了摄像机能看到的视野范围。游戏中常设为 90 到 110 度。
*   **预计算遮挡查询**：一种场景渲染优化技术。将场景划分为多个标准网格区域，预先计算好每个区域之间是否互相可见。渲染时，若当前相机所在的区域“看不见”某目标区域，则直接跳过该目标区域内物体的绘制。
*   **Intel GPA (Graphics Performance Analyzers)**：一款强大的图形性能分析工具，可以在不需要游戏源码的情况下，通过“劫持”运行中的 PC 游戏，抓取帧数据并分析其底层的渲染过程。

## 三、 工程经验与避坑指南

### 1. 行业与职业发展建议
*   **AI 的冲击**：图形学和图像处理正受到 AI 的强烈侵蚀，入门门槛被降低，导致岗位减少和内卷加剧。
*   **就业出口**：除了传统的头部游戏引擎公司，目前大量的图形学需求集中在**地图软件（高德、百度等）**以及**车企的车机系统渲染**。
*   **新兴热点**：基于 WebGL 的在线协同办公（如腾讯文档界面渲染）以及 XR 平台的 UI 渲染，因其对性能极度敏感，正成为图形学优化的新方向。

### 2. 性能优化与开发规范
*   **数据类型降级**：在图形学中，**性能永远是不充裕的**。能用 `short` 绝不用 `int`，能用 `half` 绝不用 `float`。
*   **禁止每帧向 VB 写数据**：若想在运行时改变物体的颜色，绝对不要采用“每帧修改顶点数据并重新 Copy 到 Vertex Buffer”的做法，这会造成极大的性能浪费。正确的做法是通过 **Shader** 在 GPU 中实时修改参数。
*   **关于抗锯齿**：在图像处理和 UI 绘制中，最基础的抗锯齿手段就是**绝对不要强行拉伸图像**（特别是放大），这会让锯齿变得极为明显。
*   **背景色与位运算陷阱**：在清除屏幕背景时，颜色通常使用十六进制（如 `0xFF`）。颜色通道大多通过**位运算（与运算）**截断，如果输入了一个越界的数值（如 256），其低8位为 `0x00`，会导致该通道直接归零变成黑色。

### 3. 环境配置与调试避坑
*   **Visual Studio 环境按需安装**：无需全盘安装组件。通常只需勾选“C++桌面开发”和“C++游戏开发”。注意根据项目版本手动在 Visual Studio Installer 中补充安装对应的 **Windows 10/11 SDK** 和特定版本的编译工具链（如 `v141`, `v143`），选错必然导致编译失败。
*   **善用关键字搜索**：在调试未知的 C++ 图形学 Demo 时，如果屏幕上弹出了特定的窗口标题或奇特字符，直接在整个工程中全局搜索该字符串，就能快速定位到核心的 `CreateWindow` 初始化代码。
*   **正反面与模型缺失**：如果在测试时发现模型少了一块，或者整个物体消失。大概率是因为**顶点缠绕顺序（顺时针/逆时针）写反了**，导致模型被底层 API 当作“背面”给剔除 (Cull) 掉了。
*   **坐标系陷阱**：不同引擎坐标系不同。例如 **Unity 使用 Y轴向上** 的坐标系，而 **Unreal Engine (UE) 使用 Z轴向上**。在跨引擎搬运模型或美术资产时，必然需要进行坐标矩阵的翻转处理。左乘与右乘写错时，物体会完全偏离预期位置，通过运行画面即可快速排查。
