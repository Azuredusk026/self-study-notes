# Unity Built-in 表面着色器自定义光照模型与NPR渲染实现

## 一、 核心知识点与原理剖析
1. **自定义表面着色器光照模型 (Surface Shader Lighting Model)**
    - 原理：将内置的 `Standard` 光照模型替换为自定义的 NPR 光照模型。
    - 实现：
        - 编写 `Lighting<Name>` 函数处理光照计算（传入 `SurfaceOutput`、视线方向、`UnityGI`）。
        - 编写 `<Name>_GI` 函数处理间接光（环境光、光照探针、阴影采样等预处理）。
        - 建议采用**前向渲染 (Forward Rendering)**，因为延迟渲染 (Deferred) 传递给光照模型的参数通道数量极其受限（需要写入 G-Buffer），难以传递丰富的 NPR 材质信息。

2. **漫反射的色调分离 (Diffuse Posterize)**
    - **直接光漫反射 (Direct Diffuse)**：
        - 基于 Lambert 模型，先求法线与光照方向的点积 `N·L`。
        - 使用 `step` 函数对 `N·L` 进行硬切，划分受光面与背光面（二阶或多阶分色）。
        - **羽化边缘**：若需要软过渡，可用 `smoothstep` 函数（或结合 `Remap` 重映射）替代纯硬切。
    - **间接光漫反射 (Indirect Diffuse / Environment Light)**：
        - 问题：如果直接在 RGB 通道上进行色调分离（如 `floor(color * n) / n`），由于三通道色彩变化率不同，会导致严重的**色相偏移 (Color Shift)**。
        - 解决方案：将颜色逻辑转入 HSV 空间，仅提取**明度 (Value)**进行色调分离。明度提取公式通常为 `max(R, max(G, B))`。
        - 实现细节：用原明度除掉色相/饱和度，再乘以色调分离后的新明度。

3. **高光的自定义与色调分离 (Specular Posterize)**
    - 区别于真实 PBR 高光，NPR 往往需要形状边界清晰、可控的“大高光斑”。
    - **去除菲涅尔项 (Fresnel)**：在修改 GGX 计算时，提取 PBR 的法线分布函数 (NDF, GGX) 和几何遮蔽函数 (Viss)，但必须**舍弃带有颜色的菲涅尔高光项 (F0)**，以免在 NPR 中产生不可控的边缘变色。
    - 利用 `step` 函数提取出固定的高光区域，防止粗糙的高光出现虚边。
    - **高光逻辑合并**：高光不应出现在受背光或阴影区域内，需要将计算出的高光结果与最高阶漫反射的 `step` 结果相乘。

4. **软阴影的色调分离与合并控制 (Shadow Posterize)**
    - 原理：提取 `UnityGI` 中 `data.atten`（代表阴影和光照衰减强度）进行 `step` 处理，将原生软阴影变成硬阴影。
    - **锯齿与色阶打架问题**：直接把硬化后的阴影乘回颜色中，会导致边缘锯齿和多余色阶。
    - **优化合并方案**：直接将“阴影逻辑”与漫反射的“最底层暗部 (Layer 0)”进行合并。如果所在区域的阴影衰减小于阈值，或者光源本身极暗，则判定为阴影区域，统一赋予 `ShadowColor`。这样避免了阴影与暗部计算冲突。

5. **点光源的距离衰减硬化 (Point Light Attenuation)**
    - 问题：点光源本身具有自然的距离衰减，会导致用 `step` 切出的边界边缘出现模糊的软边，破坏 NPR 风格。
    - 解决方案：将光源的衰减信息/光源颜色的明度，提前乘进 `N·L` 的结果中，使其一并参与 `step` 色调分离运算。最后在叠加光源颜色时，除掉光源明度影响（仅保留光源的色相和饱和度），以维持锐利的边界。

6. **描边技术原理与实现 (Outline)**
    - **边缘的分类**：
        - 边界边缘 (Border Edge)：只有一个面使用的边。
        - 折痕边缘 (Crease Edge)：两个面夹角大于某阈值，分为脊边和谷边。
        - 轮廓边缘 (Silhouette Edge)：共享边的两面，一面朝向观察者，一面向背。
        - 材质边缘 (Material Edge)：材质不同导致的边。
    - **法线外扩描边法 (Normal Push)**：
        - 在 `CGPROGRAM` 外另起一个 Pass，设置剔除正面渲染背面（`Cull Front`），将顶点沿法线方向向外微推。
        - **透视矫正 (Perspective Correction)**：防止描边近处极粗、远处极细，需将外推粗细系数乘以裁剪空间的 `w` 分量 (`clip space w`)，这代表了透视缩放比例，使描边在屏幕空间保持固定的像素粗细。
        - **硬边断线问题**：对于立方体等法线不连续（硬边）模型，直接推法线会导致面与面裂开。可以通过使用模型中心点向外的向量（即 `world pos - world center`）来代替法线进行外推（对凸多边形尤为有效）；或者用外推方向与世界法线做点积校验，来修正错误的法线推向。

7. **素描手绘风 NPR (Pencil Sketch) 实现**
    - 思路：基于 ASE (Amplify Shader Editor) 的 `Multi Pass Unlit` 模板自建光照循环，利用受光强度决定手绘排线纹理的显隐度——越暗的地方，素描线条出现得越密集、越清晰。
    - **解决屏幕空间采样的“淋浴门效应”**：
        - 如果直接用屏幕空间坐标 (Screen Position) 采样素描纹理，在相机或物体移动时，物体表面的素描线会像隔着玻璃一样相对屏幕不动，造成强烈的“游泳感”。
        - **三向采样技术 (Tri-planar Mapping)**：分别从世界空间的 XY、YZ、XZ 三个面上对素描纹理进行投影采样，再利用顶点的世界法线作为混合权重（`falloff` 决定过渡的软硬），将这三张结果混合起来，确保笔触固定在物体自身的世界空间结构上。

## 二、 重点术语与概念解析
- **色调分离 (Posterize)**：将连续渐变的数值或颜色（如 0~1 的浮点数）通过数学手段（如 `step` 或取整运算）强制划分为若干个离散的色阶边界，是卡通渲染最核心的技法。
- **Ramp 图 (渐变纹理)**：利用一张一维从暗到亮的渐变纹理图，根据 `N·L` 数值采样该图直接得出对应色阶颜色，用以替代复杂的数学 `step` 计算，方便美术直观调节。
- **HSV 空间提取明度 (Value)**：HSV 即色相、饱和度、明度。对带有颜色的环境光进行色阶处理时，只能动明度（V），强行操作 RGB 会造成不可控的偏色。
- **淋浴门效应 (Shower Door Effect)**：在使用屏幕空间坐标作为纹理 UV 时，物体运动会呈现出犹如在压花玻璃后移动般的漂浮感，破坏物体自身体积的附着感。
- **水密模型 (Watertight Model)**：即全封闭、没有缺口的 3D 模型，其每条边都必定由两个相邻的三角形共享。
- **脊边与谷边**：折痕边缘的两种形态。相对于平均法线向外凸起的角叫脊边，向内凹陷的叫谷边。

## 三、 工程经验与避坑指南
- **Surface Shader 中添加额外 Pass 的位置限制**
  表面着色器 (Surface Shader) 虽然看上去没有显式的 Pass 定义，但其 `CGPROGRAM` 代码在编译时会被 Unity 自动展开为前向渲染、延迟渲染、阴影投射等大约 6-7 个 Pass。如果要增加法线外扩描边，必须把新的描边 Pass 写在 `CGPROGRAM ... ENDCG` 块**外部**，否则无法生效。
- **舍弃 PBR 菲涅尔控制高光**
  直接复用 PBR 公式时一定要当心菲涅尔项（F0）。由于 NPR 需要百分百控制光影的色块和形态，非金属表面微弱的菲涅尔反射光会导致暗部凭空多出难以控制的边缘光，因此在自建 NPR 高光时应将其移除，只采用法线分布与遮蔽函数计算形状。
- **修改 ASE (Amplify Shader Editor) 代码的坑与正确做法**
  - ASE 本质是节点生成代码。如果手动用文本编辑器改了生成后的 `.shader` 文件代码，下一次在 ASE 面板一操作就会被直接覆盖洗掉。
  - **避坑法 1**：在文本编辑器中删掉文件底部附带的全部 ASE Node 注释数据，彻底将其转为纯手写 Shader。
  - **避坑法 2（推荐）**：使用 Shaderlab 的 `UsePass` 命令。在 ASE 面板左侧属性栏的 `Additional Use Passes` 里，填入包含手写描边 Pass 的 Shader 路径（如 `ShaderName/PassName`），配合开启属性参数的 `Auto Register` 功能，即可在不破坏节点树的前提下外挂手写代码。
- **插件版本与 Unity 引擎更迭的兼容性断裂**
  新版 ASE（针对 Unity 6 和 URP 设计）在老版 Built-in 管线中，使用诸如 `Light Color`、`Light Attenuation` 节点时可能导致编译器报错或无法读取光源数据。如必须在旧管线支持多光源，必须使用老版本的 ASE 插件，或手写 `Custom Expression` 节点通过内置的 `UnityShaderVariables` 强行抓取光源变量。
