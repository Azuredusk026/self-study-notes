# ACT框架优化、剧情编辑器设计与通用状态机实现

## 一、 核心知识点与原理剖析

### 1. 现存框架Bug修复
在大型ACT框架开发中，逻辑时序和泛型处理极易产生隐蔽Bug，本次课程重点修复了两个关键问题：

#### 1.1 场景关闭时的对象清理异常 (Cleanup Exception)
*   **现象描述**：关闭场景时控制台报红，提示 "some objects were not clean up"。
*   **根本原因**：在脚本的 `OnDestroy` 生命周期方法中调用了 `Rebind`（重连/重生）逻辑。由于 `Rebind` 会生成新对象，而 Unity 在销毁场景时会依次销毁所有对象，这导致了“销毁->生成->销毁”的死循环或在非法时机生成对象。
*   **修复方案**：在 `OnDestroy` 执行重连逻辑前，必须**检查当前游戏状态**。若游戏正在退出或场景正在关闭，则停止触发重生逻辑。

#### 1.2 输入数据丢失问题 (Generic Input Assignment)
*   **现象描述**：玩家按下移动键但角色无反应，调试发现 `MovementSample` 中的 `InputData` 始终为 0（默认值）。
*   **根本原因**：在父类的泛型方法 `GetInputData<T>` 中，判断 `m_InputData` 为空时直接返回了 `new T()`，但**未将其赋值给内部成员变量**。导致后续逻辑获取的始终是新实例而非被持续维护的数据实例。
*   **修复方案**：
    ```csharp
    // 错误逻辑：return new T();
    // 正确逻辑：
    m_InputData = new T(); 
    return (T)m_InputData; 
    ```

### 2. 框架架构逻辑复盘
学习复杂框架应遵循“逻辑理顺 -> 类结构细化 -> 时序管理”的步骤。

*   **数据流向**：`PlayerInputBase` (获取输入) -> `InvokerBase` (发送指令) -> `RequestReceiver` (接收请求) -> `RequestHandler` (处理队列) -> `RequestBase` (自我管理/执行) -> `BehaviourBase` (执行具体行为)。
*   **核心组件**：
    *   **PlayerBase**：作为实体基类，维护所有实例的运行顺序，严格管理**时序问题**。
    *   **独立模块**：状态机 (`StateHandler`)、移动模块 (`Movement`)、输入设备适配 (`InputDevice`) 均设计为**可插拔式模块**，不强依赖于核心请求链路。

### 3. 剧情编辑器 (Plot Editor) 设计方案
针对不同项目需求，剧情编辑器的架构设计差异巨大：

#### 3.1 运行时编辑器 (Runtime) vs Unity编辑器 (Editor)
*   **Runtime Editor**：基于游戏内 UI（如 UGUI）开发。**优点**：门槛低（普通 Unity 开发经验即可），可打包给玩家做联创工具。**缺点**：底层功能（如坐标系、网格绘制）需手写，开发量大。
*   **Unity Editor**：基于 `UnityEditor` API 开发。**优点**：可直接调用引擎底层 UI 组件（如搜索框、折叠菜单），功能强大。**缺点**：学习成本高，无法随游戏打包。

#### 3.2 逻辑结构
*   **线性/主支线结构**（如 RPG）：适合使用 `List` 维护节点。
*   **树形/分支结构**（如 GalGame）：适合使用树结构，对数据结构的存储要求更高。
*   **触发机制**：进场景、退场景、交互对象（基于事件中心 `EventCenter` 广播）。

### 4. 通用状态机 (State Machine) 实现
一套基于 `ScriptableObject` (SO) 的轻量化状态机，强调解耦与复用。

#### 4.1 状态类 (State)
*   **SO 架构**：状态作为 SO 文件存在，便于策划配置。
*   **生命周期方法**：
    *   `Enter(params)`：支持带参数进入，例如从不同状态切换至“跳跃”时，赋予不同的初始力。
    *   `Update / FixUpdate / LateUpdate`：同步 Unity 生命周期。
    *   `Exit`：清理状态。
    *   `Register / Unregister`：处理状态与状态机之间的绑定关系。

#### 4.2 状态机类 (StateMachine)
*   **核心字段**：`DefaultState` (默认状态)、`CurrentState` (当前状态)、`LastState` (上一状态)、`RegisterStateList` (已注册列表)。
*   **核心逻辑**：
    *   **初始化**：遍历“预注册列表”，将 SO 实例化（副本化）后存入运行时列表。
    *   **切换逻辑 (SwitchState)**：执行当前状态的 `Exit` -> 更新 `LastState` -> 执行新状态的 `Enter`。

## 二、 重点术语与概念解析

*   **ScriptableObject (SO)**：Unity 提供的一种用于存储大量独立于脚本实例的数据容器。在本项目中用于存储状态配置（如跳跃高度、移动速度）。
*   **Input Buffering (输入缓冲/预输入)**：为了提升手感（Feel），在角色落地前 0.1s 按下跳跃键，系统会记录该输入并在落地的瞬间执行跳跃，避免因操作过早导致的“按键失效”。
*   **Coyote Time (土狼时间)**：角色离开平台边缘后的极短时间内仍允许跳跃。
*   **I Agent (智能体接口)**：状态机所属的实体接口。状态机必须依赖一个 Agent（如玩家或AI）才能运行，确保状态逻辑能访问到物理组件。
*   **Event Center (事件中心)**：一种观察者模式的实现，用于剧情触发或状态变更的解耦合广播。

## 三、 工程经验与避坑指南

1.  **ScriptableObject 运行时保护**：
    *   **风险**：如果在运行时直接修改 SO 的属性，会导致项目资源文件被永久修改。
    *   **避坑**：在状态机初始化时，使用 `Instantiate(soAsset)` 创建一个**副本**，在运行时只修改副本数据。
2.  **UI 自动布局的递归陷阱**：
    *   在开发剧情编辑器的 Timeline（时间线）UI 时，若实现“拖拽节点自动顶开后续节点”的功能，需警惕递归逻辑导致的死循环，确保节点位置计算有明确的边界条件。
3.  **编辑器易用性设计**：
    *   程序不仅要考虑逻辑，更要考虑**策划体验**。例如：重叠节点标红、自动吸附功能、基于 Inspecter 的属性编辑窗口，这些都能极大提高生产效率。
4.  **状态切换的参数传递**：
    *   不要只做一个无参的 `SwitchState()`。在复杂动作游戏中，状态切换往往伴随着能量、速度或动画参数的传递，预留 `object[] params` 是必要的。
5.  **解耦思维**：
    *   状态机和剧情编辑器应尽量减少对具体业务类的依赖。通过接口（Interface）和事件（Event）进行通信，使得这套系统可以无缝迁移到其他项目。
