### 第一课：Win32 编程基础、底层架构与 COM 组件模型

---

### 1. 游戏引擎的底层架构层级

在实际渲染管线中，我们的代码到达 GPU 之前需要经过多个“胶水层”（接口层）。理解这些层级对于调试和优化至关重要。

* **GPU 驱动层**：最底层的硬件通信。
* **图形 API 层**：Vulkan, D3D11/12, OpenGL, Metal。每个 API 内部也有跨平台接口层，用于适配不同显卡的驱动。
* **跨平台渲染胶水层 (RHI)**：即 Render Hardware Interface。商业引擎（如虚幻）中明确存在并可修改；Unity 中被封装在底层 C++ 中，不可见。自研引擎通常封装较浅，需要技术美术 (TA) 或客户端程序直接调用 Vulkan 或 D3D 原生 API。
* **跨平台窗口胶水层**：如 GLFW, SDL。负责在不同操作系统（Win32, Linux, Mac, 甚至 Android 全屏背后的窗口系统）中创建和管理渲染目标窗口。

### 2. 内存、汇编与程序运行机制

理解 CPU 如何执行代码有助于后续深入 Shader（如 HLSL 编译出的 DXBC 字节码）的优化。

* **核心内存区域**：
* **寄存器 (Registers)**：CPU 内速度最快的数据存储，如“运行指针”记录当前执行的代码行。
* **程序内存 (Program Memory)**：存放被翻译为汇编指令的代码。
* **栈内存 (Stack)**：极快，用于存放局部变量。函数调用时申请，退出时必须按顺序（后进先出）清理。
* **堆内存 (Heap)**：通过 `new` 手动申请的内存。


* **函数调用约定 (Calling Conventions)**：
* 决定了函数跳转时参数如何入栈、返回值如何处理，以及**由谁（调用者还是被调用者）负责清理栈内存**。
* `__cdecl`：C/C++ 静态函数的默认约定。
* `__stdcall`：Windows 系统接口（WINAPI）默认使用的约定。如果调用约定写错，会导致栈清理混乱，进而引发内存错误。

### 3. Win32 程序入口与窗口创建

使用 Visual Studio 创建 Windows 窗口程序时，必须将链接器的子系统 (Subsystem) 从 Console 改为 Window。

#### 核心入口点：`wWinMain`

不再使用标准的 `main`，而是使用 Windows 专用的宽字符入口点：
| 参数名 | 描述 |
| :--- | :--- |
| `hInstance` | 当前程序的实例句柄。即编译后的 PE（可执行文件）加载到虚拟内存中的**基址**。 |
| `hPrevInstance` | 16位系统的历史遗留产物，现已废弃不用。 |
| `pCmdLine` | 启动参数（命令行参数），如快捷方式里添加的 `-console` 等。`PWSTR` 代表指向宽字符 (UTF-16) 字符串的指针。 |
| `nCmdShow` | 操作系统建议的窗口初始显示状态（如最大化、隐藏等，本质是枚举整数）。 |

#### 注册与创建窗口

1. **定义窗口类 (`WNDCLASSEX`)**：一个结构体，相当于窗口的模板，定义背景色、小图标以及对应的**消息处理函数 (`WndProc`)**。
* *注意*：必须初始化/构造，否则结构体内会充满内存垃圾。
* 使用 `RegisterClassEx` 将其注册到系统。


2. **窗口样式 (Window Styles)**：使用**位掩码 (Bitmask)** 和按位或 (`|`) 操作符组合功能。
* `WS_OVERLAPPEDWINDOW`：顶层重叠窗口（默认包含标题栏、菜单、边框、缩放按钮等）。
* `WS_POPUP`：弹出窗口（如确认对话框，会阻挡底层窗口响应）。
* `WS_CHILD`：子窗口（如资源管理器左右分栏里的独立模块）。


3. **创建与显示**：使用 `CreateWindowEx` 创建句柄 (`HWND`)，再通过 `ShowWindow` 将其显示在屏幕上（通常在图形 API 初始化完毕后调用，避免白屏闪烁）。

### 4. 消息循环与事件驱动架构

GUI 程序不能像命令行那样顺序等待输入，而是采用事件驱动模型。

* **消息 (`MSG`)**：包含消息标识符（发生了什么，如键盘按下）和相关参数。
* **消息队列**：
* **排队消息**：如鼠标移动、按键。进入队列，程序空闲时异步处理。
* **非排队消息**：如窗口激活。由操作系统直接插队调用目标线程的窗口过程函数，不进入循环。


* **阻塞式消息循环**：
* `GetMessage`：从队列抓取消息。队列为空时会**阻塞当前线程**，降低功耗。遇到 `WM_QUIT` 返回 0，循环结束。
* `TranslateMessage`：将键盘按键消息（如按了 `A` 键）翻译为字符消息（输入了字符 `'a'`）。
* `DispatchMessage`：将消息分发给对应窗口的 `WndProc` 处理。

### 5. 窗口过程函数 (`WndProc`) 与输入交互

负责接收并响应系统派发来的消息。所有未自定义处理的消息应交由 `DefWindowProc` 兜底处理（如拖动窗口边框）。

* **参数解析 (`WPARAM`, `LPARAM`)**：通常是 32 位整数，经常被拆分为高 16 位 (`HIWORD`) 和低 16 位 (`LOWORD`) 使用。
* **窗口生命周期消息**：
* `WM_CLOSE`：轻柔关闭（如点击右上角 X）。可被拦截（例如弹出一个 MessageBox 确认是否退出）。
* `WM_DESTROY`：强制销毁窗口。必须在处理中调用 `PostQuitMessage(0)` 发送 `WM_QUIT` 给线程，否则窗口消失但线程仍在后台死循环。


* **键盘交互 (`WM_KEYDOWN`)**：
* `WPARAM`：虚拟键码 (`VK_...`)。字母和数字键需通过十六进制偏移量计算（如 `keycode - 0x41` 判断字母）。
* `LPARAM` 高 16 位：位掩码标志（判断是否按了右侧扩展 Ctrl、是否按住 Alt）。
* `LPARAM` 低 16 位：连击次数（Repeat Count，按住键不放产生的合并消息）。


* **鼠标交互 (`WM_LBUTTONDOWN` 等)**：
* 利用 `windowsx.h` 中的 `GET_X_LPARAM` 宏，提取鼠标相对于工作区左上角的局部坐标。
* *进阶提示*：开发自由视角的 3D 游戏需使用“原始输入技术 (Raw Input)”，普通窗口局部坐标会受限于屏幕边缘。

### 6. COM 组件与对象模型 (Component Object Model)

COM 是一套跨语言的二进制接口标准，是 Windows 提供底层 API（包括 DirectX、UI 交互等）的核心方式。

* **核心概念**：
* **CLSID**：COM 类的全局唯一标识符 (GUID)。
* **IID**：接口的全局唯一标识符。
* **机制**：客户端不能直接 `new` 实例化对象或访问其内部数据。必须通过传入 CLSID 告诉服务端创建对象，同时传入 IID 获取该对象上对应的虚函数接口指针。


* **初始化与清理**：程序启动时必须调用 `CoInitializeEx`，退出前调用 `CoUninitialize` 防止内存/服务泄漏。
* **实例化流程示例 (打开文件对话框)**：
1. 使用 `CoCreateInstance` 创建 `FileOpenDialog` 对象，并请求 `IFileOpenDialog` 接口（通常搭配 `IID_PPV_ARGS` 宏自动匹配接口 ID）。
2. 调用 `Show()` 阻塞线程等待用户选择文件。
3. 通过 `GetResult()` 获取 `IShellItem`（虚拟文件对象）指针。
4. 调用 `GetDisplayName()` 拿到文件系统路径字符串。


* **内存管理**：
* 接口指针：不要手动释放。引入 `<wrl.h>` 使用智能指针 `Microsoft::WRL::ComPtr<T>`，它会自动管理引用计数 (`Release`)。
* COM 内部分配的内存（如返回的字符串路径）：必须使用 `CoTaskMemFree` 手动释放。
