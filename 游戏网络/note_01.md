# 联机游戏开发实战：从局域网原理到 Mirror 插件应用

## 一、 核心知识点与原理剖析

### 1. 联机游戏的分类与架构基础
*   **匹配机制（Matching Logic）：** 在进入正式游戏前，玩家通常在主服务器上创建房间（Room）。系统根据玩家分段、延迟、地区等策略将玩家分配至同一房间，以优化延迟体验。
*   **局域网模式（LAN Mode）：**
    *   **架构：** 采用 **主机（Host）+ 客机（Client）** 模式。
    *   **主机（Host）：** 既是服务端（Server）又是客户端（Client）。负责保存整局游戏的所有数据，运行所有核心逻辑。若主机掉线，整局游戏结束。
    *   **客机（Client）：** 仅作为接收者，向服务端发送指令并接收服务端信息来执行本地表现。若客机掉线，不影响其他玩家。
    *   **优点：** 技术成本与资金成本极低，适合大学生做 Demo 或集训项目。

### 2. 网络通信的三种基本行为
*   **单播（Unicast）：** 主机仅向特定的某一个客机发送信息。
    *   *案例：* 斗地主发牌，主机告知玩家 A 其手牌为“黑桃 A”，玩家 B 不知情；RTS 游戏中的战争迷雾（Fog of War），仅告知玩家其视野内的信息。
*   **广播（Broadcast）：** 主机向所有连接的客机同步相同的信息。
    *   *案例：* 某玩家打出一张牌，所有玩家的屏幕上都需要显示该出牌动作。
*   **指令（Command）：** 客机向主机发起行为请求。
    *   *逻辑：* 客机不能直接修改游戏数据。例如客机想出牌，需发送“请求出牌”指令给主机，由主机校验通过后再广播结果。

### 3. 数据校验与防作弊（Anti-Cheat）
*   **作弊本质：** 客户端向服务器发送了虚假或异常的数据（如修改位移速度、修改 Buff 数值、自瞄等）。
*   **鉴别方式：**
    *   **位置队列检测：** 记录坐标队列，若单次位移超过阈值则判定异常。
    *   **行为轨迹分析：** 利用机器学习分析玩家操作（如鼠标移动轨迹）是人类行为还是算法行为。
    *   **扫盘/撞库：** 检测系统运行进程中是否存在黑名单中的外挂程序。
    *   **逻辑校验：** 服务器保存玩家状态（如手牌列表），若玩家尝试出一张其不存在的牌，服务器将予以驳回并刷新其本地数据。

### 4. 权威模式（Authority）
*   **服务器权威（Server Authority）：** 绝大多数数据的变更必须经过服务器判定合法后才生效。安全性最高，但存在响应延迟。
*   **客户端权威（Client Authority）：** 客户端直接执行操作并通知服务器。响应快，但极易导致数据错乱（如两人同时拾取同一个箱子里的唯一装备）。
*   *折中策略：* 在 UI 交互等实时性要求高的环节使用客户端本地处理，在关键数据交互（如打开箱子、扣除血量）时加入服务器校验或独占标记。

## 二、 重点术语与概念解析

### 1. 传输协议（Transport Protocols）
*   **TCP：** 安全性高，有三次握手/四次挥手机制，但速度较慢。适合对实时性要求不高、数据必须准确的卡牌游戏。
*   **UDP：** 速度极快，但不保证包的顺序与完整性，容易丢包。
*   **KCP：** 介于 TCP 与 UDP 之间的算法实现。它在 UDP 的基础上通过算法保证了可靠性，且延迟比 TCP 低。**Mirror 默认推荐 KCP**。

### 2. Mirror 核心组件与功能
*   **NetworkManager：** 联机引擎的总入口。
    *   **Scene Management：** 配置在线/离线场景切换。
    *   **Transport：** 挂载通信协议（如 KCP Transport）。
    *   **Player Prefab：** 注册玩家控制的角色预制体。
    *   **Registered Spawnable Prefabs：** 注册所有需要在网络中动态生成的物体（如子弹、球、道具）。
*   **NetworkIdentity：** 物体的“身份证”。任何需要网络同步的对象都必须挂载此组件，用于标识物体的唯一性及所有权。
*   **NetworkTransform：** 用于同步位置（Position）、旋转（Rotation）和缩放（Scale）。
    *   *优化建议：* 仅勾选必要的同步项（如只同步坐标）以节省带宽。
*   **NetworkBehaviour：** 脚本基类。继承此类后可使用以下特殊标记：
    *   **`[Command]`：** 客户端调用，在服务端运行。方法名建议以 `Cmd` 开头。
    *   **`[ClientRpc]`：** 服务端调用，在所有客户端运行。方法名建议以 `Rpc` 开头。
    *   **`[SyncVar]`：** 同步变量。当服务器修改此变量时，所有客户端会自动更新。
    *   **Hook（钩子函数）：** 当 `SyncVar` 变量变化时自动触发的方法，常用于 UI 刷新或动画切换。

### 3. 开发辅助工具
*   **PararSync：** 多开工具。允许在 Unity 编辑器中克隆项目，实现“主项目开发，克隆项目实时同步测试”，避免频繁打包。

## 三、 工程经验与避坑指南

### 1. 架构设计：单机与联机代码解耦
*   **核心建议：** 不要尝试在同一套脚本中兼容单机与联机。建议通过继承体系将两者分开。
    *   *案例（阿比亚项目）：* 创建 `AbiaEntity` 基类处理基础逻辑，派生出 `SinglePlayer`（继承 `MonoBehaviour`）和 `NetworkPlayer`（继承 `NetworkBehaviour`）。
*   **优势：** 降低维护难度。当单机关卡更新或联机模式调整时，互不干扰，仅在必要时同步功能接口。

### 2. 同步策略：指令同步 vs 状态同步
*   **误区纠正：** 联机游戏不会每帧同步所有角色的骨骼旋转数据。
*   **实操做法：** 仅同步 **指令（Key Events）**。例如玩家按下攻击键，客户端发送“攻击请求”，服务器校验后下发“执行攻击逻辑”广播，各客户端在本地播放动画并计算判定。

### 3. 常见 Bug 避坑
*   **丢包导致的状态偏差：** 如大炮旋转角度不一。原因通常是同步了“速度+时间”而非“最终旋转角度”。对于关键表现，建议直接同步最终的四元数（Quaternion）。
*   **本地玩家判定：** 在 `Update` 等逻辑中，务必使用 `isLocalPlayer` 判断，防止操作自己的按键导致场景中所有角色同步动作。

### 4. 游戏性与平衡补丁
*   **开发者心态：** 不要站在玩家的对立面。刻意针对玩家的“跳关”或“无限连”套路进行恶意砍伐（Nerf）会导致评价大幅下滑（如《杀戮尖塔 2》补丁事件）。
*   **Demo 选型：** 初学者建议从简单的 PVP 射击、火柴人大乱斗或吸血鬼类幸存者（同步击杀、同步血量）入手，降低逻辑耦合复杂度。
