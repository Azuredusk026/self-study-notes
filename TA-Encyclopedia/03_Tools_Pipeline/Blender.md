# Blender

## 一句话定义
Blender 是集建模、动画、雕刻、几何节点和渲染于一体的 DCC 工具。

## 为什么需要它

TA 需要理解 `Blender`，因为它会影响资源制作、引擎配置、画面表现、调试路径或团队协作边界。把它写成明确条目，可以减少口头经验传递，并让问题排查有稳定入口。

## 核心原理

- 输入：DCC 场景、模型、贴图、命名规则、导出配置、版本库和引擎导入设置。
- 处理过程：通过脚本、插件、批处理或 CI 检查执行规范校验、格式转换、批量处理和报告输出。
- 输出：可导入引擎的资产、检查报告、修复建议、导出包或团队规范文档。
- 所在层级：DCC / Editor Tool / Pipeline。

## 技术美术中的典型用途

- 减少手工导出和命名错误。
- 把资产规范变成可执行检查。
- 连接美术、程序、构建和版本管理流程。

## Unity 中的相关场景

常见于 Editor Tool、AssetPostprocessor、导入预设、Addressables/AssetBundle 构建前检查和资源报告。

## Unreal Engine 中的相关场景

常见于 Editor Utility Widget、Python、Data Validation、导入脚本、批量重定向和内容浏览器整理流程。

## 与其他概念的区别

| 概念 | 区别 |

| [[DCC工具链]] | DCC 工具链偏制作软件侧；Pipeline 更强调跨软件、跨引擎和团队流程。 |
| [[资源检查工具]] | 资源检查工具是管线中的一个执行节点，不等同于完整管线设计。 |

## 常见误区

1. 只记概念名，不确认它在项目中的输入、输出和所在管线阶段。
2. 把引擎默认效果当成固定标准，忽略渲染管线、平台和项目配置差异。
3. 没有保留可复现的测试场景，导致问题只能靠截图或主观描述沟通。

## 面试可能怎么问

### 问题 1

`Blender` 解决的核心问题是什么？

回答要点：先说明它在工具链与生产管线中处理哪类输入和输出，再结合一个项目场景说明为什么需要它。

### 问题 2

在 Unity 和 Unreal 中落地 `Blender` 时，TA 需要分别关注什么？

回答要点：比较两边的工具入口、资源规则、调试方式和平台限制，不要只背 API 名称。

### 问题 3

如果 `Blender` 相关效果或资产在项目中出问题，你会怎么排查？

回答要点：从资源输入、引擎配置、运行时状态、性能指标和最小复现场景逐层缩小范围。
