### 第三课：HLSL 高级着色器语言基础与 GPU 资源管理

---

### 一、 渲染管线演进与着色器语言生态

在深入代码之前，我们需要了解当前图形 API 的生态：

* **管线演进史**：
* **DX9/DX10**：基础的顶点/片元着色。能做到的极限画面大约是《孤岛危机3》。
* **DX11**：引入了**计算着色器 (Compute Shader)** 和曲面细分 (Tessellation)，是目前绝大多数非光追 3A 游戏的主力标准。
* **DX12**：引入了底层硬件控制和硬件光线追踪 (Raytracing)。


* **着色器语言之争**：
* **GLSL**：伴随 OpenGL，主要用于老旧移动端、开源平台。正在被淘汰。
* **Metal Shading Language (MSL)**：苹果生态独占，完全隔离。
* **HLSL**：微软开发。如今随着 Vulkan 的普及，HLSL 可以被编译为 SPIR-V 字节码在 Vulkan 上运行，因此 **HLSL 已成为目前全平台图形开发（除苹果外）的绝对主流**。

### 二、 HLSL 基本数据类型与运算规则

HLSL 可以看作是“没有指针的 C 语言”，但为了适应图形计算，它扩展了许多特定的数据结构。

#### 1. 标量 (Scalars)

* 支持常规的 `bool`, `int`, `uint`, `float`, `double`（GPU 通常不支持 64 位整数 `long`）。
* **特殊浮点数**：`unorm` (无符号规范化，范围 0~1) 和 `snorm` (带符号规范化，范围 -1~1)。在 CPU 端通常传入 0-255 的整数，GPU 会自动将其解释为 0.0~1.0 的浮点数。
* **移动端优化 (`half`)**：16位浮点数。在 PC 端通常直接作为 32 位 `float` 处理，但在移动端使用 `half` 可以极大降低 GPU 发热与耗电。

#### 2. 向量 (Vectors)

最多支持四维向量：`float2`, `float3`, `float4` (或泛型写法 `vector<float, 4>`)。

* **访问方式**：可以通过 `xyzw`（空间坐标表示）或 `rgba`（颜色表示）访问，两者完全等效。也可以使用数组下标 `[0]`, `[1]` 访问。
* **分量混用 (Swizzling)**：HLSL 允许极度自由的向量重组。例如 `float3 v2 = v1.xzz;` 或追加分量 `float4 v3 = float4(v1.xyz, 1.0);`。
* **计算规则**：GPU 中的向量乘法 `v1 * v2` 是**逐分量相乘**（x乘x，y乘y），而不是数学意义上的点积或叉积！(点积需使用 `dot()` 函数)。

#### 3. 矩阵 (Matrices)

如 `float4x4` 或 `float2x3`（两行三列）。

* 提取元素：推荐使用两层数组索引 `m[row][col]` 的方式提取矩阵中的标量，更符合直觉。

### 三、 HLSL 内置函数与调试技巧

千万不要自己手写基础数学实现（如开方、三角函数），HLSL 的内置函数在 GPU 底层有专门的指令优化。

* **数值范围控制**：
* `max(a,b)` / `min(a,b)`：求最大最小值。
* `clamp(x, min, max)`：将 x 限制在 min 和 max 之间。
* `saturate(x)`：极其常用，等效于 `clamp(x, 0.0, 1.0)`。常用于颜色输出前的防溢出限制。
* `lerp(a, b, t)`：线性插值。`smoothstep`：Hermite 插值（两端导数为0，过渡更平滑，无突变亮线）。


* **向量计算**：`length(v)`（求长度）、`distance(a,b)`（求距离）、`normalize(v)`（归一化）。
* **特殊函数**：`atan2(y, x)`，常用于计算屏幕坐标到中心的极坐标夹角，是制作“旋转/雷达扫描”特效的核心函数。
* **异常调试函数**：
* 在 GPU 中一旦发生**除以 0** 或对负数开方，会产生 `Inf`（无穷大）或 `NaN`（非数字），导致整个屏幕闪烁或黑/白屏。
* 使用 `isinf(x)` 和 `isnan(x)` 捕获错误，并在 Shader 中强行 `return float4(1,0,1,1);`（输出亮粉色）来定位 Bug。

### 四、 语义 (Semantics) 系统机制

HLSL 中变量的命名并不重要，**语义才是连接 CPU 与 GPU、以及各个 Shader 阶段的唯一桥梁**。

* **输入装配器 -> 顶点着色器 (VS)**：
* `POSITION` (位置), `COLOR` (颜色), `NORMAL` (法线)。
* `TEXCOORD0` ~ `TEXCOORDn`：通常用于传递 UV 坐标，但行业约定俗成用它来传递**任何自定义数据**。


* **顶点着色器 -> 片元着色器 (PS)**：
* 必须且只能有一个 `SV_POSITION`（系统值：裁剪空间坐标，光栅化后变为屏幕像素坐标）。
* 其他插值数据通过 `TEXCOORD` 或 `COLOR` 传递。

### 五、 GPU 三大核心资源与 CPU 绑定代码

为了让 Shader 动起来，必须从 CPU (C#) 传递数据进 GPU 显存。

#### 1. 常量缓冲区 (Constant Buffer / cbuffer)

用于存放单次 DrawCall 中不会改变的全局参数（如：鼠标位置、时间、MVP变换矩阵）。

* **HLSL 端**：使用 `cbuffer` 声明，寄存器使用 `register(b0)`。
* **数据对齐**：常量缓冲区在 GPU 中严格按照 16 字节 (128位，即 `float4`) 跨度对齐。必须在结构体末尾填充无用变量（Padding）补齐位数。
* **C# 上传方式 (Dynamic + Map)**：
1. 使用 `Usage = ResourceUsage.Dynamic` 和 `CPUAccessFlags = CpuAccessFlags.Write` 创建 Buffer。
2. 每一帧调用 `DeviceContext.Map()` 获取显存虚拟地址指针。
3. 在 C# 中利用 `Span<T>` 将结构体数据安全地复制进显存，然后 `Unmap()`。
4. 使用 `VSSetConstantBuffers` 绑定到管道。



#### 2. 纹理 (Texture2D) 与 着色器资源视图 (SRV)

* **纹理格式**：CPU端通常不需要自己读取 PNG/JPG 拼装字节，而是利用 Nvidia Texture Tools 将图片转换为显卡原生支持的压缩格式（如 **DDS 格式，BC7 压缩**）。
* **视图机制**：GPU 管线不能直接读取纹理内存。必须为纹理创建一个 **Shader Resource View (SRV)**。
* **HLSL 端**：使用 `Texture2D<float4>` 声明，寄存器使用 `register(t0)`。可以使用 `tx.Load()` 直接通过整型像素坐标读取。

#### 3. 采样器 (SamplerState)

定义如何根据 UV 坐标（0~1）从纹理中“提取”颜色。

* **过滤模式 (Filter)**：
* `Point` (最近点采样)：直接取距离中心最近的像素颜色。放大后有明显的马赛克方块感（适合像素风游戏）。
* `Linear` (线性采样 / 双线性插值)：在相邻的四个像素间进行平滑过渡。


* **寻址模式 (Address Mode)**：当 UV 坐标超出 0~1 范围时如何处理。
* `Wrap` (Repeat)：图像无限重复平铺。
* `Mirror`：重复平铺，但每次重复都会镜像翻转。
* `Clamp`：边缘像素向外无限拉伸。


* **HLSL 端**：使用 `SamplerState` 声明，寄存器使用 `register(s0)`。使用 `tx.Sample(sampler, uv)` 获取颜色。

### 六、 Bonus: LDR 与 HDR 色彩

* 我们在 Shader 中计算输出的颜色通常在 `[0, 1]` 之间，大于 1 的部分会被显示器或纹理格式截断（Clamp）到 1，这称为 **LDR（低动态范围）**。
* 如果将渲染目标的格式设置为支持浮点数，且显示器支持，则可以保留并显示大于 1 的颜色亮度（如现实中的太阳光、高亮特效），这就是 **HDR（高动态范围）** 渲染的基础。
